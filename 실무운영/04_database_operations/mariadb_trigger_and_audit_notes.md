# mariadb_trigger_and_audit_notes

## 1. 개요
MariaDB에서 트리거 권한 조정과 감사 로그 설정을 수행할 때 확인했던 항목을 정리한 문서입니다.

## 2. 작업 배경
운영 중 트리거 생성/수정이 필요하거나, 특정 사용자 행위를 감사 로그로 남겨야 하는 요구가 발생할 수 있습니다.
이 경우 권한, DEFINER, 바이너리 로그 정책, 로그 저장 경로를 함께 봐야 했습니다.

## 3. 문제/요구사항
- 트리거 생성 권한 부족
- `DEFINER` 호스트 정보가 실제 실행 환경과 맞지 않음
- 감사 로그를 파일로 남기되 회전/보관 기준이 필요

## 4. 판단 및 선택 이유
- 트리거는 단순 쿼리 생성보다 계정/호스트/복제 정책 영향을 함께 봐야 함
- Audit Log는 설치 여부보다 어떤 이벤트를 남기고, 어떤 사용자를 포함/제외할지가 핵심
- 운영 로그는 무한정 쌓지 않고 회전 정책을 같이 가져가야 함

## 5. 적용 절차 요약
### Trigger
- `log_bin_trust_function_creators` 상태 확인
- 대상 사용자에 `TRIGGER` 권한 부여
- `DEFINER`를 실제 접속 호스트 기준으로 조정

### Audit
- `server_audit` 플러그인 설치
- 감사 이벤트 범위 설정
- 파일 경로, 회전 크기, 회전 개수 설정
- 포함/제외 사용자 지정
- 감사 로깅 활성화

## 6. 검증 방법
- 트리거 생성/수정 쿼리 재실행
- 트리거 동작 테스트
- 감사 로그 파일 생성 여부 확인
- 로그 로테이션 정책 반영 확인

## 7. 결과 및 운영 효과
- 운영 DB 변경 이력과 보안 감사 요구를 동시에 대응할 수 있음
- 단순 SQL 메모가 아니라 운영 제약 조건까지 드러나는 문서로 바꿀 수 있음
