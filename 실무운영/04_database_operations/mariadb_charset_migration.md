# mariadb_charset_migration

## 1. 개요
MariaDB 환경에서 데이터베이스/테이블 문자셋이 혼재할 때,
전체 백업/복원과 서버 기본값 변경을 어떤 순서로 검토했는지 정리한 문서입니다.

## 2. 작업 배경
운영 DB 일부가 `latin` 계열 문자셋으로 남아 있으면 한글 데이터 처리나 이관 시 문제가 발생할 수 있습니다.
특히 전체 덤프/복원 과정에서 옵션 조합에 따라 결과가 달라질 수 있어 사전 검증이 필요했습니다.

## 3. 문제/요구사항
- DB/테이블 문자셋이 일관되지 않음
- 백업 시 `--default-character-set`, `--skip-set-charset` 적용 여부에 따라 결과 차이 발생
- 서버 기본 설정과 실제 데이터 상태가 다를 수 있음

## 4. 판단 및 선택 이유
- 문자셋 변경은 운영 중 즉시 반영보다 **백업-복원 검증**이 우선
- 한 번에 전체 전환할지, DB/테이블 단위로 나눌지 판단이 필요
- 설정값만 바꾸면 해결될 것처럼 보이지만 실제 데이터가 이미 다른 인코딩이면 복원 결과가 달라질 수 있음

## 5. 검토한 시나리오
- 백업 시 `--default-character-set=utf8` 적용
- 서버 설정을 UTF 계열로 변경 후 일반 백업/복원
- `--skip-set-charset` 옵션을 이용한 백업/복원
- 임포트 전 DB 전체 삭제 후 재적재

## 6. 검증 방법
- DB/테이블/컬럼 문자셋 조회
- 샘플 한글 데이터 조회
- 애플리케이션 연결 후 깨짐 여부 확인
- 덤프 파일 내부 `SET NAMES` / charset 관련 구문 확인

## 7. 결과 및 운영 효과
- 문자셋 이슈는 단순 설정값 변경보다 데이터 상태 점검이 우선임을 문서화할 수 있음
- 실제 운영에서 마주치는 정합성 이슈를 보여줄 수 있어 설치 가이드보다 포트폴리오 가치가 높음

## 8. 운영 체크포인트
- `utf8`와 `utf8mb4`를 구분해서 설계할 것
- 전체 백업/복원 전에 샘플 DB로 반복 검증
- 애플리케이션/JDBC 연결 문자셋도 함께 확인
