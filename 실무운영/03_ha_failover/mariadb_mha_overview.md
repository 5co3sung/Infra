# mariadb_mha_overview

## 1. 개요
MariaDB Master-Slave 기반 환경에서 MHA를 통해 장애 전환을 구성하려는 흐름을 정리한 문서입니다.

## 2. 작업 배경
DB 계층은 단순 백업만으로는 충분하지 않고, 장애 시 빠른 승격과 복제 재구성이 필요할 수 있습니다.
특히 운영 환경에서는 장애 감지와 승격 절차를 표준화할 필요가 있습니다.

## 3. 환경
- Rocky Linux 계열
- MariaDB 10.11.x
- MHA 0.58
- Manager 1대 + DB Node 다수

## 4. 문제/요구사항
- 복제 계정, MHA 관리 계정, 서버 ID 등 기본 조건이 선행되어야 함
- Master/Slave 복제가 정상이어야 MHA가 의미를 가짐
- 장애 전환 후 데이터 정합성과 재동기화 절차까지 고려해야 함

## 5. 판단 및 선택 이유
- 수동 승격보다 운영 절차를 줄일 수 있음
- 복제/장애전환이 잦거나 운영인력이 제한된 환경에서 이점이 있음
- 다만 설치 자체보다, 사전 복제 품질과 장애 시나리오 검증이 더 중요함

## 6. 적용 절차 요약
1. 복제 계정 생성
2. MHA 관리 계정 생성
3. `server.cnf`에서 `server_id`, `log-bin`, `read_only` 등 역할별 설정
4. Slave 복제 시작 및 상태 확인
5. OS 계정 생성
6. MHA Manager 패키지/Perl 모듈 설치
7. Failover 시나리오 점검

## 7. 검증 방법
- `show master status`
- `show slave status\G`
- MHA 체크 명령으로 노드/복제 상태 확인
- Master 장애 가정 후 전환 시나리오 리허설

## 8. 결과 및 운영 효과
- DB 장애조치 절차를 운영자 경험에만 의존하지 않게 정리 가능
- 'MHA를 설치했다'보다 '왜 MHA가 필요한 환경이었는가'를 설명할 수 있음

## 9. 운영 체크포인트
- MHA는 복제가 불안정하면 효과가 떨어짐
- 계정, SSH 신뢰, 로그 보존, VIP/DNS 연동 여부를 별도로 검토
- Failover 이후 애플리케이션 연결 문자열 변경 필요 여부 확인
