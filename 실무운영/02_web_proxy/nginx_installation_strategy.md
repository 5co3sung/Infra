# nginx_installation_strategy

## 1. 개요
Nginx를 신규 구축할 때 단순 설치 명령이 아니라, 어떤 저장소/버전을 선택할지와 이후 운영에 어떤 영향을 주는지를 정리한 문서입니다.

## 2. 작업 배경
웹 계층에서 Reverse Proxy, HTTPS 종료, 정적 리소스 제공, 업스트림 분산 같은 역할을 담당할 프론트 서버가 필요했습니다.
배포판 기본 저장소 버전보다 최신 기능 또는 운영 표준에 맞는 버전을 선택해야 할 필요도 있었습니다.

## 3. 환경
- Rocky Linux 계열
- Nginx 1.25 계열 검토
- Reverse Proxy / Upstream 구성 예정 서버

## 4. 문제/요구사항
- 기본 OS 저장소 버전만으로는 원하는 버전 정책을 맞추기 어려움
- Stable/Mainline 중 어떤 트랙을 사용할지 판단 필요
- 설치 이후 upstream, SSL, 보안 설정이 이어질 수 있도록 표준 구조가 필요

## 5. 판단 및 선택 이유
- 공식 저장소를 별도로 등록하면 버전 관리 기준을 통일하기 쉬움
- 기능 검증이 우선이면 stable, 최신 기능 검토가 필요하면 mainline을 선택할 수 있음
- 단순 설치보다 이후 `conf.d`, upstream, 로그, systemd 운영까지 연결되는 구조를 먼저 보는 것이 중요함

## 6. 적용 절차
1. `yum-utils` 설치
2. 공식 Nginx 저장소 파일 구성
3. stable / mainline 중 사용할 트랙 선택
4. 패키지 설치 및 서비스 등록
5. 기본 설정 파일 구조 확인
6. upstream 또는 reverse proxy 설정 반영

### 업스트림 예시 관점
- WAS 여러 대를 하나의 upstream으로 묶어 프록시할 수 있음
- 프론트에서 요청을 받아 백엔드 포트 구조를 숨길 수 있음

## 7. 검증 방법
- `nginx -v`, `nginx -t`
- `systemctl status nginx`
- 기본 포트 응답 확인
- upstream 대상 정상 분산 여부 확인

## 8. 결과 및 운영 효과
- 웹 계층 표준화가 가능해짐
- 백엔드 직접 노출을 줄이고 프록시 구조를 일관되게 가져갈 수 있음
- 이후 HTTPS, 헤더 제어, 접근 통제 문서와 연결되는 기반 문서가 됨

## 9. 운영 체크포인트 / 트러블슈팅
- 공식 저장소 등록 시 배포판 버전과 경로 호환성을 먼저 확인
- upstream만 설정하고 헬스체크/타임아웃 정책을 빠뜨리지 않도록 주의
- `nginx -t` 검증 없이 재기동하지 않기
