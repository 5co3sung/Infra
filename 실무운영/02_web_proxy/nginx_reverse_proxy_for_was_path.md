# nginx_reverse_proxy_for_was_path

## 1. 개요
Nginx를 통해 WAS 내부 특정 경로를 외부에 직접 노출하지 않고 제어한 구성을 포트폴리오형으로 재정리한 문서입니다.

## 2. 작업 배경
사용자는 특정 웹 경로에 접근해야 했지만, 애플리케이션 서버 포트를 직접 외부에 노출하는 방식은 운영과 보안 측면에서 부담이 있었습니다.
따라서 웹 계층에서 요청 경로를 통제하고, 백엔드 서비스는 내부 포트로만 유지하는 구조가 필요했습니다.

## 3. 환경
- Front: Nginx Web Server
- Back: WAS 또는 내부 Nginx
- 특정 URL Path만 프록시 대상에 전달

## 4. 문제/요구사항
- 특정 디렉터리 또는 경로만 외부 접근 허용 필요
- 실제 백엔드 포트와 내부 경로를 외부에 노출하고 싶지 않음
- 원본 요청 헤더와 클라이언트 IP를 백엔드에 전달해야 함

## 5. 판단 및 선택 이유
### 고려한 대안
- WAS 포트를 직접 외부에 오픈
- 웹서버 없이 애플리케이션에서 직접 라우팅
- Nginx reverse proxy 사용

### 최종 선택
Nginx reverse proxy를 사용해 경로 단위로 접근을 제어하는 방식을 선택했습니다.

### 선택 이유
- 접근 경로를 웹 계층에서 일원화 가능
- `X-Forwarded-*`, `Host`, `X-Real-IP` 같은 헤더 전달이 용이
- 백엔드 포트와 인증서 처리를 프론트 계층에서 분리 가능

## 6. 적용 절차
1. 프론트 Nginx `location` 블록 정의
2. `proxy_pass` 대상 지정
3. `proxy_set_header`로 원본 요청 정보 전달
4. 백엔드에서 동일한 경로 구조 또는 alias/root 정리
5. 설정 검증 후 재기동

## 7. 검증 방법
- 프론트 URL로 접근 시 백엔드 경로가 정상 노출되는지 확인
- `access.log`, `error.log`로 프록시 전달 여부 확인
- 백엔드에서 `Host`, `X-Forwarded-For` 헤더 확인
- `proxy_pass`의 trailing slash 여부에 따른 경로 해석 확인

## 8. 결과 및 운영 효과
- 백엔드 직접 노출 없이 필요한 URL 경로만 제공 가능
- 웹 계층에서 접근정책, 인증서, 헤더 처리를 통합 관리할 수 있음
- 향후 ACL, 인증, 리다이렉트 정책을 프론트에서 확장하기 쉬워짐

## 9. 운영 체크포인트 / 트러블슈팅
- `location` 경로와 백엔드 실제 경로를 다르게 잡으면 404가 발생하기 쉬움
- `proxy_pass` 끝의 `/` 유무에 따라 라우팅 결과가 달라질 수 있음
- HTTPS 백엔드로 프록시하는 경우 인증서 신뢰 정책도 함께 검토 필요
