# nginx_header_hardening

## 1. 개요
보안 점검에서 `Server: nginx` 헤더 노출을 줄이기 위해 검토한 방식과, 실제 운영 관점에서 어떤 접근이 적절한지 정리한 문서입니다.

## 2. 작업 배경
보안 취약점 점검 과정에서 응답 헤더에 서버 정보가 노출되는 항목을 조치 요청받았습니다.
단순히 '숨긴다'가 목적이 아니라, 기존 운영 방식에 영향을 최소화하면서 대응 가능한 방법이 필요했습니다.

## 3. 문제/요구사항
- 응답 헤더에 Nginx 서버 정보가 노출됨
- 가능한 한 서비스 영향 없이 조치하고 싶음
- 패키지 설치형 Nginx에서 바로 해결되지 않는 경우가 있음

## 4. 판단 및 선택 이유
### 고려한 대안
- 기본 설정만으로 `server_tokens off` 적용
- 추가 모듈 없이 그대로 유지
- `headers-more-nginx-module`을 사용해 응답 헤더 제어

### 선택 관점
- 버전 정보 노출만 줄일 목적이면 `server_tokens off`가 먼저 검토 대상
- 헤더 자체를 더 세밀하게 제어하려면 추가 모듈 또는 빌드 방식이 필요
- 다만 운영 중인 패키지 기반 Nginx를 소스 재컴파일로 바꾸면 관리 복잡도가 증가하므로, 효과와 비용을 함께 봐야 함

## 5. 적용 절차 요약
- 모듈 사용이 필요한 경우 Nginx 소스와 `headers-more-nginx-module` 확보
- `./configure` 시 `--add-module` 옵션으로 빌드
- `make`, `make install`
- systemd 서비스 파일 정비
- 헤더 제거/변경 규칙 적용

## 6. 검증 방법
- `curl -I`로 응답 헤더 확인
- 재기동 후 서비스 정상 응답 확인
- 패키지 기반 운영 절차와 충돌 여부 확인

## 7. 결과 및 운영 효과
- 보안 점검 대응에 필요한 헤더 제어 방안을 확보
- 단순 설정 조치와 소스 빌드 조치의 차이를 비교할 수 있게 됨
- 문서상으로는 '숨김'보다 '운영 비용과 변경 영향 판단'이 중요함을 보여줄 수 있음

## 8. 운영 체크포인트 / 트러블슈팅
- 소스 빌드 전환 시 패키지 업데이트 체계가 달라짐
- 기존 Nginx 사용자/경로/systemd 설정을 다시 맞춰야 할 수 있음
- 헤더 숨김만으로 보안이 완료되는 것은 아니며, 버전/패치 관리가 병행되어야 함
