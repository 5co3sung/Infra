# rocky_linux_baseline_and_hardening

## 1. 개요
리눅스 서버를 업무에 투입하기 전 공통적으로 적용하던 기본 설정을 정리한 문서입니다.
단순 설치 절차보다, 운영 안정성과 보안 기준을 먼저 맞추는 관점으로 작성했습니다.

## 2. 작업 배경
새 서버를 투입할 때마다 계정 정책, SSH 설정, 파일 핸들 수, 세션 타임아웃, 방화벽/SELinux 상태를
각각 확인해야 했고, 이 기준이 일관되지 않으면 장애 대응과 보안 점검 시 편차가 생길 수 있었습니다.

## 3. 환경
- Linux 계열 서버(Ubuntu / Rocky Linux 중심)
- 신규 구축 또는 이관 직후 서버
- 운영/개발 서버 공통 적용 가능한 베이스라인

## 4. 문제/요구사항
- 서비스별로 OS 초기 설정 기준이 달라 관리 포인트가 분산됨
- root 직접 접속, 세션 유지시간, limits 설정 등이 서버마다 다르게 적용될 수 있음
- 운영 전에 최소 보안 기준과 성능 기본값을 맞출 필요가 있음

## 5. 판단 및 선택 이유
- 서버 구축 초기에 공통 설정을 먼저 적용해야 이후 미들웨어 설정이 단순해짐
- SSH 정책과 계정 정책은 보안성뿐 아니라 작업 이력 관리 측면에서도 중요함
- `limits.conf`, `sysctl.conf`, `TMOUT` 같은 값은 실제 운영 중 프로세스/접속 안정성에 직접 영향이 있음

## 6. 적용 절차
### 공통 OS 설정
- SELinux 상태 확인 및 운영 기준에 맞게 조정
- 방화벽 정책 확인 후 필요 포트만 허용
- `sysctl.conf`로 네트워크/파일 핸들 관련 값 정리
- `limits.conf`로 `nofile`, `nproc` 상향

### 계정/접속 정책
- 서비스용 일반 계정 생성
- 필요한 경우 sudo 권한을 명시적으로 부여
- `sshd_config`에서 root 직접 로그인 제한
- `TMOUT` 설정으로 유휴 세션 자동 종료

### 사용자 편의/운영성
- 언어/인코딩 설정 정리
- shell profile, alias, PATH 정리
- 필요 런타임(JDK, Python 등) 설치

## 7. 검증 방법
- `getenforce`, `systemctl status firewalld`, `sysctl -p` 결과 확인
- `ulimit -n`, `ulimit -u` 값 확인
- 신규 일반 계정으로 SSH 접속 테스트
- root 직접 로그인 차단 여부 확인
- `java -version`, `python3 --version` 등 런타임 검증

## 8. 결과 및 운영 효과
- 서버별 기본 상태를 예측 가능하게 관리할 수 있음
- 보안 점검 전 사전 정비 시간을 줄일 수 있음
- 서비스 설치 문서보다 먼저 봐야 하는 공통 베이스라인 역할을 할 수 있음

## 9. 운영 체크포인트 / 트러블슈팅
- root 로그인 차단 전, 대체 접속 계정으로 로그인 가능한지 반드시 확인
- SELinux/방화벽 비활성화는 편의성이 높지만 운영 기준과 보안 요구사항을 함께 검토해야 함
- `limits.conf`만 수정하고 재로그인/재부팅 없이 값이 안 바뀌는 경우가 있음
