# openssl_upgrade_notes

## 1. 개요
기본 패키지 버전으로 요구사항을 충족하기 어려운 환경에서 OpenSSL을 소스 또는 수동 방식으로 올릴 때의 주의사항을 정리했습니다.

## 2. 작업 배경
애플리케이션 또는 보안 요구사항 때문에 기본 배포판에 포함된 OpenSSL 버전보다 높은 버전이 필요할 수 있습니다.
이 경우 단순 업그레이드보다 라이브러리 링크와 패키지 의존성 영향까지 함께 고려해야 했습니다.

## 3. 문제/요구사항
- 패키지 저장소 버전만으로는 필요한 OpenSSL 버전을 맞추기 어려움
- `/usr/bin/openssl`, `libssl.so`, `libcrypto.so` 링크가 꼬이면 기존 명령이나 패키지 관리가 영향을 받을 수 있음

## 4. 판단 및 선택 이유
- 시스템 기본 OpenSSL을 대체할 때는 설치보다 **롤백 가능성**을 먼저 고려해야 함
- 심볼릭 링크 변경은 빠르지만 패키지 의존성에 영향을 주므로 검증 범위를 넓게 잡아야 함
- 운영 중인 서버라면 테스트 서버에서 동일 절차를 먼저 검증하는 편이 안전함

## 5. 적용 절차 요약
- 빌드 도구와 의존 라이브러리 설치
- 소스 압축 해제 후 `./config`, `make`, `make install`
- `ld.so.conf.d`에 라이브러리 경로 추가
- `ldconfig` 실행
- `openssl` 바이너리 및 라이브러리 심볼릭 링크 갱신
- PATH 반영 및 버전 확인

## 6. 검증 방법
- `openssl version`
- `find / -name libssl.so*`
- 실제 애플리케이션 기동 테스트
- `yum`, `dnf` 등 패키지 관리자 동작 여부 확인

## 7. 운영 체크포인트 / 트러블슈팅
- 라이브러리 링크 변경 후 패키지 관리 명령이 실패할 수 있음
- `libssl.so` 경로 불일치로 기존 서비스가 뜨지 않는 경우가 있음
- 소스 설치 시 제거 절차(`make uninstall`, 링크 복원)를 반드시 준비해야 함
